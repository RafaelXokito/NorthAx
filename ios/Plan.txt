## 2. ✅ Workout Effort Graph (Block Visualization)

**Goal:** Show a horizontal timeline graph of a workout's blocks, where:
- X axis = time (0 → total workout duration)
- Y axis = effort metric relevant to the sport (pace for running, HR/power zone for cycling, etc.)
- Each block is rendered as a colored segment at its target intensity

**Sport-specific intensity axis:**
- Running → pace (min/km or min/mile)
- Cycling → heart rate zone OR power zone (depending on user's settings preference)
- Swimming → pace per 100m
- (Other sports follow the same pattern — use whichever metric is configured)

**Example (running workout):**
  [0–5 min]  Warmup       → 5:40 min/km  (low zone, e.g. blue)
  [5–20 min] Threshold    → ~4:10 min/km  (high zone, e.g. red/orange)
  [20–35 min] Zone 3      → ~4:40 min/km  (medium zone, e.g. yellow)
  [35–40 min] Cooldown    → 5:40 min/km  (low zone, e.g. blue)

**Tasks:**
- ✅ Design a `WorkoutEffortGraphView` SwiftUI component that takes an array of workout blocks.
- ✅ Each block has: name, duration (minutes), target intensity value, sport type.
- ✅ Render blocks as adjacent rectangles scaled by duration on X, coloured by zone/intensity on Y.
- ✅ Add the graph to the Plan view (per-session detail) and/or the Coach view session card.
- ✅ Ensure the intensity metric displayed respects the user's preferred unit (km vs mile, HR vs power).
- ✅ Support the graph being tappable per block to show block details.

## 6. ⬜ Plan Generation Rate Limit (2×/day)

**Goal:** Prevent users from regenerating their plan more than twice per calendar day, to avoid abuse and encourage deliberate plan changes.

**Rules:**
- A user may generate (or regenerate) their plan at most 2 times per calendar day (midnight-to-midnight in the user's local timezone).
- The limit resets at midnight local time.
- Both explicit regenerations ("Rebuild my plan") and first-time generation count toward the limit.
- Changing training schedule / enrolled sports does NOT auto-trigger generation — it only prompts the user, who then chooses to regenerate (and that counts toward the limit).

**Backend:**
- ⬜ Track `plan_generations_today` and `last_plan_generation_date` on the athlete profile (or a dedicated audit table).
- ⬜ The plan generation endpoint returns HTTP 429 with a body like `{"error": "plan_limit_reached", "resets_at": "<ISO timestamp of next midnight>"}` when the limit is exceeded.

**iOS:**
- ⬜ Before showing the "Generate Plan" / "Rebuild Plan" button/action, fetch (or use cached) remaining generation count for today.
- ⬜ Display remaining attempts near the generate action: e.g. "1 of 2 generations remaining today".
- ⬜ When the limit is hit, disable the button and show a message: "You've used both plan generations for today. Come back tomorrow to rebuild your plan."
- ⬜ Show the reset time in the user's local timezone: "Resets at midnight (2 Jul)".
- ⬜ Handle the 429 response gracefully in case the local count is stale.

**Tasks:**
- ⬜ Add `planGenerationsToday: Int` and `planGenerationResetAt: Date` to the athlete profile model / API response.
- ⬜ Update `NorthAxAPI` / `PlanEngine` to surface the 429 and parse the `resets_at` field.
- ⬜ In `TrainingPlanView` (and anywhere else a regeneration can be triggered), gate the action behind the remaining count.
- ⬜ Show a remaining-count badge / label next to the generate button.
- ⬜ Write a unit test: attempting a third generation within the same day returns the correct error and the UI disables the button.

## 7. 🟡 Main Page — Plan-Centric Layout with "Week at a Glance"

**Goal:** Make the main (Dashboard) screen plan-first. The dominant element is the current week's training plan with live completion tracking, cross-referenced against workouts already imported from intervals.icu (and Apple Health / Garmin).

### Week at a Glance (top of the main page)

A compact horizontal strip showing all 7 days of the current week (Mon–Sun). Each day shows:
- The sport icon(s) planned for that day (or a rest icon if no session planned).
- A completion state per session: Planned / Done / Missed / Rest.
  - **Done** = a matching completed workout was found in the imported data (intervals.icu / Garmin / Apple Health).
  - **Missed** = the planned day has passed and no matching workout was found.
  - **Planned** = scheduled in the future (or today, not yet done).
  - **Rest** = no session scheduled.
- Today is highlighted. Tapping a day scrolls to that session card below.

### Session Cards (below the strip)

The rest of the main page lists the week's session cards in order. Each card shows:
- Sport, planned date, target duration and intensity summary.
- Completion badge (Done / Missed / Planned).
- If **Done**: show the actual workout's key stats pulled from intervals.icu (actual distance, avg HR, avg pace/power, TSS/load) alongside the planned targets so the user can see how closely they matched.
- If **Planned** and it is today: no "Start session" CTA — the card is tappable and opens the dedicated Workout Detail View (see below).
- The `WorkoutEffortGraphView` (feature #2) is NOT embedded inline in the card — it lives in the Workout Detail View to keep the card compact.
- Each card is always tappable regardless of completion state.

### Workout Detail View

A full-screen push (or large sheet) that opens when any session card is tapped. Contains all information about that planned session:

1. **Header** — sport icon, session title, planned date, completion badge.

2. **Workout Effort Graph** — the full `WorkoutEffortGraphView` (feature #2) showing the block structure with sport-appropriate intensity axis. Tapping a block shows a popover with block details (name, duration, target pace/power/HR).

3. **Planned targets summary** — total duration, total distance (if applicable), estimated load/TSS, target intensity distribution (e.g. "15 min threshold, 15 min Z3, 10 min easy").

4. **Actual vs Planned** (shown only when the session is matched to a completed workout):
   - Side-by-side table: Planned vs Actual for duration, distance, avg pace/power, avg HR, load.
   - A second effort graph showing the *actual* workout intensity over time (from intervals.icu data) overlaid or alongside the planned graph, so the athlete can see how closely they executed the plan.

5. **Activity graphs** (shown only for completed sessions, data sourced from integrations):
   - **Heart Rate over time** — full HR trace for the workout duration. Show HR zones as coloured background bands so the athlete can see time-in-zone at a glance.
   - **Pace over time** (running/swimming) — actual pace trace vs the planned pace target line.
   - **Power over time** (cycling, if power data available) — actual power trace vs planned target, with FTP reference line.
   - **Elevation profile** (running/cycling, if GPS data available) — gradient area chart.
   - **Cadence over time** (running/cycling, if available) — line chart.
   - Each graph is only rendered if its data is actually present in the matched workout — no empty or zeroed-out charts.
   - Graphs are scrollable vertically within the view; the section header reads "Activity Data" with a source badge (intervals.icu / Garmin / Apple Health).
   - Data sourced from whichever integration reported the matched workout (intervals.icu first, Garmin fallback, Apple Health last).

6. **Workout notes** — free-text field the athlete or coach attached to the session (read-only for now).

7. **Source** — small badge indicating where the matched workout data came from (intervals.icu / Garmin / Apple Health).

### Workout Matching Logic

Matching a planned session to an actual imported workout:
- Match on: same calendar day + same sport type.
- If multiple workouts exist for the same day/sport, pick the one closest in duration to the planned session.
- Store the match result (`matchedWorkoutId`) on the planned session in the backend so it is stable across syncs.
- Expose a "Unlink" action if the auto-match is wrong, letting the user manually link or unlink a workout.

### Data Flow

- `AthleteStore` loads the current `WeeklyPlan` from the API.
- It also fetches the week's imported workouts from intervals.icu (and Apple Health as fallback) for the matching date range.
- The matching logic runs client-side in a `PlanMatchingEngine` (or server-side and returned as part of the plan response — TBD).
- The matched state is cached locally and refreshed whenever the store syncs.

## 9. 🟡 Daily Pre-fetched AI Switch Suggestions

**Goal:** At the start of each day, proactively request AI-generated alternative session suggestions from the backend — one per planned workout that day. This replaces the current on-demand deterministic switcher with context-aware AI recommendations that are ready before the user even opens the app.

### When to fetch

- Also triggered on **first app launch of the day** (checked against a stored `lastSuggestionFetchDate`; if that date ≠ today, fetch immediately on load).
- If the fetch fails (offline, timeout), the app falls back to the existing deterministic `switchSuggestion(for:)` engine — no empty state. And stating that the generated data were not inteligent.

### What to request

- For each **planned session today**, fire one independent request to the backend.
  - e.g. if today has a Run + Strength session → 2 requests, one per session.
- Each request sends the full context payload (see below) so the AI can reason about the athlete's state.

### Context payload per request

Each suggestion request includes:
- **Planned session being considered** — sport, target duration, intended intensity, workout blocks.
- **Today's readiness metrics** — HRV, resting HR, sleep score, sleep duration, body weight (all available sources, resolved per `MetricSourcePriority`).
- **Previous 7 days training load** — per-day ATL contribution, sport breakdown (from intervals.icu / Apple Health / manual entries).
- **Remaining weekly plan** — the other sessions planned for the rest of the week (sport + intensity), so the AI can factor in what's coming up and avoid overloading a muscle group before a hard session two days later.
- **Athlete thresholds** — FTP, LTHR, max HR, pace zones — so suggestions are personalised to the athlete's fitness level rather than generic.
- **Enrolled sports + sport-specific preferences** — cycling target (HR vs power), running unit (km/mile), muscle group split for strength.

### Response

The backend returns a ranked list of 2–4 alternative sessions, each with:
- Sport, title, duration, intensity label, description.
- A short AI rationale (1–2 sentences) explaining why this alternative fits today's context.
- Estimated training load (so the user can see it stays load-equivalent).
- For Strength: full exercise list with sets/reps.
- For running/cycling/swimming: workout blocks (compatible with `WorkoutEffortGraphView`).

### iOS tasks

- ✅ Add `dailySuggestions: [PlannedSessionID: [SwitchSuggestion]]` to `AthleteStore` — keyed by the planned session so each card shows its own alternatives.
- ✅ Store `lastSuggestionFetchDate` in UserDefaults; compare to today on app foreground.
- ✅ In `WorkoutDetailView`, show the pre-fetched alternatives as a "Switch to..." section. If suggestions aren't ready yet, show a loading shimmer; if fetch failed, show the deterministic fallback silently.
- ✅ The AI rationale text is shown per suggestion card.
- ✅ Cache suggestions locally so they survive a foreground/background cycle without re-fetching.
- ✅ Invalidate the cache and re-fetch if: the user's readiness metrics update significantly (>10% HRV change), or the weekly plan is regenerated.
- ⬜ Register a `BGAppRefreshTask` for midnight fetch (best-effort; OS may delay it).

### Switch rate limit (2×/day)

- ⬜ The user may apply a switch suggestion at most **2 times per calendar day** (midnight-to-midnight local time), across all sessions combined.
- ⬜ **Backend:** track `session_switches_today` and `last_switch_date` on the athlete profile. The apply-switch endpoint returns HTTP 429 with `{"error": "switch_limit_reached", "resets_at": "<ISO timestamp>"}` when exceeded.
- ⬜ **iOS:** track remaining switches locally (`switchesRemainingToday: Int`, default 2, persisted in UserDefaults with a date key). Decrement on each successful switch.
  - ⬜ In `WorkoutDetailView`, when `switchesRemainingToday == 0`, show the suggestions list as greyed-out with a lock badge and message: "You've used both activity switches for today. Come back tomorrow."
  - ⬜ Show remaining count near the "Switch to..." section header: e.g. "Switch to... (1 of 2 remaining today)".
  - ⬜ Handle the backend 429 gracefully in case the local count is stale (re-sync from error response).

## 11. ✅ Week Navigation in "Week at a Glance"

**Goal:** Both the Dashboard (main page) and the Plan page "Week at a Glance" block should let the user browse past and future weeks with a simple, consistent navigation control embedded directly in the block — no separate picker strip.

### Constraints

- **Past:** unlimited — the user can scroll back through any historical week.
- **Future:** max 2 weeks ahead of the current week (the AI plan only generates 2 weeks forward).
- Navigation must be identical in both Dashboard and Plan page so the experience is consistent.

### UI Design

The navigation lives in the block header row, replacing the plain "WEEK AT A GLANCE" label:

```
◀  Week of 30 Jun – 6 Jul  ▶
```

- Left `◀` arrow: always enabled for past weeks; the right `▶` arrow is disabled (greyed) when at +2 weeks.
- The centre label shows the week date range (Mon–Sun). When offset = 0 it reads **"This week"**, ±1 reads **"Last week"** / **"Next week"**, ±2 reads **"2 weeks ago"** / **"In 2 weeks"**.
- Arrows are `Button` with `.axTertiary` style when disabled, `.white` when enabled.
- **Swipe gesture** on the strip: swipe left = next week, swipe right = previous week (capped at +2). Uses a `DragGesture` with a 50pt threshold to avoid conflicting with the outer scroll view.

## 13. ⬜ Strava Integration + Cross-Source Activity Data Merging

**Goal:** Add Strava as a second supported sport-activity integration. For users without intervals.icu, Strava provides activity data (GPS, pace, HR, cadence, elevation). When both integrations are active, merge activity data field-by-field using a per-field source priority, so no data is discarded — whichever source has a field the other lacks, that field is used.

### Why Strava

- Strava is the most widely used sports tracking platform — a large share of recreational athletes log there exclusively.
- intervals.icu has deeper analytics (CTL/ATL/TSB, wellness, structured workouts), but Strava has broader adoption.
- The priority model ensures Strava supplements intervals.icu rather than conflicting with it.

### OAuth connect flow

- Credentials: `STRAVA_CLIENT_ID` and `STRAVA_CLIENT_SECRET` stored in backend environment variables only — never in client code or committed files.
- Flow: standard OAuth 2.0 with PKCE. The backend holds the refresh token; the iOS app triggers connect via a backend-mediated redirect (same pattern as intervals.icu).
- Scopes required: `activity:read_all` (to read all activities including private ones).
- Add Strava as a card in `IntegrationsView` (currently in the "Coming soon" section — promote it to "Connected" once implemented).

### Data fetched from Strava

- Activities list: id, type, start date, elapsed time, distance, moving time, average HR, max HR, average cadence, average watts (if power meter), total elevation gain, start lat/lng.
- Activity streams (per activity): time, heartrate, velocity_smooth, cadence, watts, altitude, latlng — same shape as intervals.icu streams, returned from `GET /activities/{id}/streams`.
- Athlete profile: VO2Max estimate (if Strava has it), FTP (if set).
- Note: Strava does NOT provide sleep data, HRV, or CTL/ATL/TSB — those remain intervals.icu–only.

### Cross-source activity data merging (field-level priority)

**Core principle:** when the same activity exists in two sources (matched by date + sport + duration), merge field-by-field. The higher-priority source wins for any field it provides; the lower-priority source fills in fields the higher source lacks.

**`ActivitySourcePriority`** — a new user-configurable ordered list of sources per data category:

| Category | Fields | Default priority |
|----------|--------|-----------------|
| Wellness / recovery | HRV, RHR, sleep, VO2Max, CTL/ATL | intervals.icu → Apple Health → Manual |
| Activity summary | distance, avg HR, avg pace, load/TSS | intervals.icu → Strava → Garmin → Apple Health |
| Activity streams | HR trace, pace trace, power, cadence, elevation | intervals.icu → Strava → Garmin |

**Merge algorithm (per matched activity):**
1. Collect all source records for the activity (intervals.icu, Strava, Garmin, Apple Health).
2. For each field, iterate the priority list; take the first non-null value.
3. Store the winning source per field in an `ActivityFieldSources` map so the UI can show provenance badges ("HR from intervals.icu", "distance from Strava").
4. For time-series streams: if the higher-priority source has a stream, use it entirely. If it lacks a stream (e.g. no power from intervals.icu but Strava has it from a power meter), use the lower-priority source's stream for that channel.

**User-configurable priority:**
- Extend `MetricPriorityView` (Settings → Integrations → Source Priority) to include an "Activity data" section alongside the existing wellness section.
- User can drag-reorder sources per category. Changes persist via the API.

### `ActivitySourcePriority` model

- New model alongside `MetricSourcePriority` — or extend it with activity-specific fields.
- Stored in the athlete profile on the backend; synced on load like `MetricSourcePriority`.
