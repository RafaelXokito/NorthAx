## 2. тЬЕ Workout Effort Graph (Block Visualization)

**Goal:** Show a horizontal timeline graph of a workout's blocks, where:
- X axis = time (0 тЖТ total workout duration)
- Y axis = effort metric relevant to the sport (pace for running, HR/power zone for cycling, etc.)
- Each block is rendered as a colored segment at its target intensity

**Sport-specific intensity axis:**
- Running тЖТ pace (min/km or min/mile)
- Cycling тЖТ heart rate zone OR power zone (depending on user's settings preference)
- Swimming тЖТ pace per 100m
- (Other sports follow the same pattern тАФ use whichever metric is configured)

**Example (running workout):**
  [0тАУ5 min]  Warmup       тЖТ 5:40 min/km  (low zone, e.g. blue)
  [5тАУ20 min] Threshold    тЖТ ~4:10 min/km  (high zone, e.g. red/orange)
  [20тАУ35 min] Zone 3      тЖТ ~4:40 min/km  (medium zone, e.g. yellow)
  [35тАУ40 min] Cooldown    тЖТ 5:40 min/km  (low zone, e.g. blue)

**Tasks:**
- тЬЕ Design a `WorkoutEffortGraphView` SwiftUI component that takes an array of workout blocks.
- тЬЕ Each block has: name, duration (minutes), target intensity value, sport type.
- тЬЕ Render blocks as adjacent rectangles scaled by duration on X, coloured by zone/intensity on Y.
- тЬЕ Add the graph to the Plan view (per-session detail) and/or the Coach view session card.
- тЬЕ Ensure the intensity metric displayed respects the user's preferred unit (km vs mile, HR vs power).
- тЬЕ Support the graph being tappable per block to show block details.

## 6. тмЬ Plan Generation Rate Limit (2├Ч/day)

**Goal:** Prevent users from regenerating their plan more than twice per calendar day, to avoid abuse and encourage deliberate plan changes.

**Rules:**
- A user may generate (or regenerate) their plan at most 2 times per calendar day (midnight-to-midnight in the user's local timezone).
- The limit resets at midnight local time.
- Both explicit regenerations ("Rebuild my plan") and first-time generation count toward the limit.
- Changing training schedule / enrolled sports does NOT auto-trigger generation тАФ it only prompts the user, who then chooses to regenerate (and that counts toward the limit).

**Backend:**
- тмЬ Track `plan_generations_today` and `last_plan_generation_date` on the athlete profile (or a dedicated audit table).
- тмЬ The plan generation endpoint returns HTTP 429 with a body like `{"error": "plan_limit_reached", "resets_at": "<ISO timestamp of next midnight>"}` when the limit is exceeded.

**iOS:**
- тмЬ Before showing the "Generate Plan" / "Rebuild Plan" button/action, fetch (or use cached) remaining generation count for today.
- тмЬ Display remaining attempts near the generate action: e.g. "1 of 2 generations remaining today".
- тмЬ When the limit is hit, disable the button and show a message: "You've used both plan generations for today. Come back tomorrow to rebuild your plan."
- тмЬ Show the reset time in the user's local timezone: "Resets at midnight (2 Jul)".
- тмЬ Handle the 429 response gracefully in case the local count is stale.

**Tasks:**
- тмЬ Add `planGenerationsToday: Int` and `planGenerationResetAt: Date` to the athlete profile model / API response.
- тмЬ Update `NorthAxAPI` / `PlanEngine` to surface the 429 and parse the `resets_at` field.
- тмЬ In `TrainingPlanView` (and anywhere else a regeneration can be triggered), gate the action behind the remaining count.
- тмЬ Show a remaining-count badge / label next to the generate button.
- тмЬ Write a unit test: attempting a third generation within the same day returns the correct error and the UI disables the button.

## 7. ЁЯЯб Main Page тАФ Plan-Centric Layout with "Week at a Glance"

**Goal:** Make the main (Dashboard) screen plan-first. The dominant element is the current week's training plan with live completion tracking, cross-referenced against workouts already imported from intervals.icu (and Apple Health / Garmin).

### Week at a Glance (top of the main page)

A compact horizontal strip showing all 7 days of the current week (MonтАУSun). Each day shows:
- The sport icon(s) planned for that day (or a rest icon if no session planned).
- A completion state per session: Planned / Done / Missed / Rest.
  - **Done** = a matching completed workout was found in the imported data (intervals.icu / Garmin / Apple Health).
  - **Missed** = the planned day has passed and no matching workout was found.
  - **Planned** = scheduled in the future (or today, not yet done).
  - **Rest** = no session scheduled.
- Today is highlighted. Tapping a day scrolls to that session card below.

### Session Cards (below the strip)

The rest of the main page lists the week's session cards in order. Each card shows:
- Sport, planned date, target duration and intensity summary.
- Completion badge (Done / Missed / Planned).
- If **Done**: show the actual workout's key stats pulled from intervals.icu (actual distance, avg HR, avg pace/power, TSS/load) alongside the planned targets so the user can see how closely they matched.
- If **Planned** and it is today: no "Start session" CTA тАФ the card is tappable and opens the dedicated Workout Detail View (see below).
- The `WorkoutEffortGraphView` (feature #2) is NOT embedded inline in the card тАФ it lives in the Workout Detail View to keep the card compact.
- Each card is always tappable regardless of completion state.

### Workout Detail View

A full-screen push (or large sheet) that opens when any session card is tapped. Contains all information about that planned session:

1. **Header** тАФ sport icon, session title, planned date, completion badge.

2. **Workout Effort Graph** тАФ the full `WorkoutEffortGraphView` (feature #2) showing the block structure with sport-appropriate intensity axis. Tapping a block shows a popover with block details (name, duration, target pace/power/HR).

3. **Planned targets summary** тАФ total duration, total distance (if applicable), estimated load/TSS, target intensity distribution (e.g. "15 min threshold, 15 min Z3, 10 min easy").

4. **Actual vs Planned** (shown only when the session is matched to a completed workout):
   - Side-by-side table: Planned vs Actual for duration, distance, avg pace/power, avg HR, load.
   - A second effort graph showing the *actual* workout intensity over time (from intervals.icu data) overlaid or alongside the planned graph, so the athlete can see how closely they executed the plan.

5. **Activity graphs** (shown only for completed sessions, data sourced from integrations):
   - **Heart Rate over time** тАФ full HR trace for the workout duration. Show HR zones as coloured background bands so the athlete can see time-in-zone at a glance.
   - **Pace over time** (running/swimming) тАФ actual pace trace vs the planned pace target line.
   - **Power over time** (cycling, if power data available) тАФ actual power trace vs planned target, with FTP reference line.
   - **Elevation profile** (running/cycling, if GPS data available) тАФ gradient area chart.
   - **Cadence over time** (running/cycling, if available) тАФ line chart.
   - Each graph is only rendered if its data is actually present in the matched workout тАФ no empty or zeroed-out charts.
   - Graphs are scrollable vertically within the view; the section header reads "Activity Data" with a source badge (intervals.icu / Garmin / Apple Health).
   - Data sourced from whichever integration reported the matched workout (intervals.icu first, Garmin fallback, Apple Health last).

6. **Workout notes** тАФ free-text field the athlete or coach attached to the session (read-only for now).

7. **Source** тАФ small badge indicating where the matched workout data came from (intervals.icu / Garmin / Apple Health).

### Workout Matching Logic

Matching a planned session to an actual imported workout:
- Match on: same calendar day + same sport type.
- If multiple workouts exist for the same day/sport, pick the one closest in duration to the planned session.
- Store the match result (`matchedWorkoutId`) on the planned session in the backend so it is stable across syncs.
- Expose a "Unlink" action if the auto-match is wrong, letting the user manually link or unlink a workout.

### Data Flow

- `AthleteStore` loads the current `WeeklyPlan` from the API.
- It also fetches the week's imported workouts from intervals.icu (and Apple Health as fallback) for the matching date range.
- The matching logic runs client-side in a `PlanMatchingEngine` (or server-side and returned as part of the plan response тАФ TBD).
- The matched state is cached locally and refreshed whenever the store syncs.

## 9. ЁЯЯб Daily Pre-fetched AI Switch Suggestions

**Goal:** At the start of each day, proactively request AI-generated alternative session suggestions from the backend тАФ one per planned workout that day. This replaces the current on-demand deterministic switcher with context-aware AI recommendations that are ready before the user even opens the app.

### When to fetch

- Also triggered on **first app launch of the day** (checked against a stored `lastSuggestionFetchDate`; if that date тЙа today, fetch immediately on load).
- If the fetch fails (offline, timeout), the app falls back to the existing deterministic `switchSuggestion(for:)` engine тАФ no empty state. And stating that the generated data were not inteligent.

### What to request

- For each **planned session today**, fire one independent request to the backend.
  - e.g. if today has a Run + Strength session тЖТ 2 requests, one per session.
- Each request sends the full context payload (see below) so the AI can reason about the athlete's state.

### Context payload per request

Each suggestion request includes:
- **Planned session being considered** тАФ sport, target duration, intended intensity, workout blocks.
- **Today's readiness metrics** тАФ HRV, resting HR, sleep score, sleep duration, body weight (all available sources, resolved per `MetricSourcePriority`).
- **Previous 7 days training load** тАФ per-day ATL contribution, sport breakdown (from intervals.icu / Apple Health / manual entries).
- **Remaining weekly plan** тАФ the other sessions planned for the rest of the week (sport + intensity), so the AI can factor in what's coming up and avoid overloading a muscle group before a hard session two days later.
- **Athlete thresholds** тАФ FTP, LTHR, max HR, pace zones тАФ so suggestions are personalised to the athlete's fitness level rather than generic.
- **Enrolled sports + sport-specific preferences** тАФ cycling target (HR vs power), running unit (km/mile), muscle group split for strength.

### Response

The backend returns a ranked list of 2тАУ4 alternative sessions, each with:
- Sport, title, duration, intensity label, description.
- A short AI rationale (1тАУ2 sentences) explaining why this alternative fits today's context.
- Estimated training load (so the user can see it stays load-equivalent).
- For Strength: full exercise list with sets/reps.
- For running/cycling/swimming: workout blocks (compatible with `WorkoutEffortGraphView`).

### iOS tasks

- тЬЕ Add `dailySuggestions: [PlannedSessionID: [SwitchSuggestion]]` to `AthleteStore` тАФ keyed by the planned session so each card shows its own alternatives.
- тЬЕ Store `lastSuggestionFetchDate` in UserDefaults; compare to today on app foreground.
- тЬЕ In `WorkoutDetailView`, show the pre-fetched alternatives as a "Switch to..." section. If suggestions aren't ready yet, show a loading shimmer; if fetch failed, show the deterministic fallback silently.
- тЬЕ The AI rationale text is shown per suggestion card.
- тЬЕ Cache suggestions locally so they survive a foreground/background cycle without re-fetching.
- тЬЕ Invalidate the cache and re-fetch if: the user's readiness metrics update significantly (>10% HRV change), or the weekly plan is regenerated.
- тмЬ Register a `BGAppRefreshTask` for midnight fetch (best-effort; OS may delay it).

### Switch rate limit (2├Ч/day)

- тмЬ The user may apply a switch suggestion at most **2 times per calendar day** (midnight-to-midnight local time), across all sessions combined.
- тмЬ **Backend:** track `session_switches_today` and `last_switch_date` on the athlete profile. The apply-switch endpoint returns HTTP 429 with `{"error": "switch_limit_reached", "resets_at": "<ISO timestamp>"}` when exceeded.
- тмЬ **iOS:** track remaining switches locally (`switchesRemainingToday: Int`, default 2, persisted in UserDefaults with a date key). Decrement on each successful switch.
  - тмЬ In `WorkoutDetailView`, when `switchesRemainingToday == 0`, show the suggestions list as greyed-out with a lock badge and message: "You've used both activity switches for today. Come back tomorrow."
  - тмЬ Show remaining count near the "Switch to..." section header: e.g. "Switch to... (1 of 2 remaining today)".
  - тмЬ Handle the backend 429 gracefully in case the local count is stale (re-sync from error response).

## 11. тЬЕ Week Navigation in "Week at a Glance"

**Goal:** Both the Dashboard (main page) and the Plan page "Week at a Glance" block should let the user browse past and future weeks with a simple, consistent navigation control embedded directly in the block тАФ no separate picker strip.

### Constraints

- **Past:** unlimited тАФ the user can scroll back through any historical week.
- **Future:** max 2 weeks ahead of the current week (the AI plan only generates 2 weeks forward).
- Navigation must be identical in both Dashboard and Plan page so the experience is consistent.

### UI Design

The navigation lives in the block header row, replacing the plain "WEEK AT A GLANCE" label:

```
тЧА  Week of 30 Jun тАУ 6 Jul  тЦ╢
```

- Left `тЧА` arrow: always enabled for past weeks; the right `тЦ╢` arrow is disabled (greyed) when at +2 weeks.
- The centre label shows the week date range (MonтАУSun). When offset = 0 it reads **"This week"**, ┬▒1 reads **"Last week"** / **"Next week"**, ┬▒2 reads **"2 weeks ago"** / **"In 2 weeks"**.
- Arrows are `Button` with `.axTertiary` style when disabled, `.white` when enabled.
- **Swipe gesture** on the strip: swipe left = next week, swipe right = previous week (capped at +2). Uses a `DragGesture` with a 50pt threshold to avoid conflicting with the outer scroll view.
- Tapping a day in the strip on the Dashboard jumps to that day's session card (or, for a non-current week, opens a compact past-week summary sheet).

### Data model per week offset

| Offset | Source | Content shown |
|--------|--------|---------------|
| < 0 (past) | intervals.icu imported activities grouped by week | Actual completed workouts only тАФ sport icon, duration, done marker. No planned targets. |
| 0 (current) | AI plan + `PlanMatchingEngine` | Planned sessions + completion state matched against imports тАФ existing behaviour. |
| +1 / +2 (future) | AI plan only | Planned sessions, all shown as "Planned" (no completion state yet). |

### Dashboard vs Plan page differences

**Dashboard:**
- The `WeekGlanceView` strip remains compact (dots + icons). For non-current weeks, the "Today's session" card below is hidden and replaced by a compact "No session data" message or a week-summary row showing total sessions completed.
- Tapping a day dot on a past week opens a lightweight `PastDaySheet` showing that day's imported workout(s) summary (sport, duration, load). No planned vs actual comparison тАФ just the raw imported data.
- Navigating away from the current week shows a subtle "тЖР Back to this week" pill below the strip.

**Plan page:**
- The existing horizontal week-tab scroller (`weekPicker`) is removed; the `WeekGlanceView` header arrows replace it.
- The sessions list below updates to show the selected week's sessions (planned for future weeks, actual imports for past weeks).
- For past weeks, session rows show actual workout data (sport, actual duration, distance if available) styled differently from planned rows (no target intensity, no effort graph block тАФ show a compact stats row instead).

### `AthleteStore` changes

- Add `func weekData(offset: Int) async -> WeekData` тАФ returns a `WeekData` struct containing days + matches for that week offset.
  - offset тЙе 0: resolved from `weeklyPlans` (already loaded).
  - offset < 0: fetches the date range from `IntervalsService` (cached by ISO week key in a local dict).
- Add `historicalWeeks: [Int: WeekData]` cache dict to `AthleteStore` so past weeks aren't re-fetched on every swipe.
